Item 23: Understand std::move and std::forward

  • std::moveperforms an unconditional cast to an rvalue. In and of itself, it doesn’t move anything.

  • std::forwardcasts its argument to an rvalue only if that argument is boound to an rvalue.
  • Neither std::move nor std::forwardfo anything at runtime.

Item 25: Use std::move on rvalue references, std::forward on universal references

  • Apply std::move to rvalue references and std::forward to universal references the last time each is used.

  • Do the same thing for rvalue references and universal references being return from functions that return by value.

  • Never apply std::move or std::forward to local objects if they would other-wise be eligible for the return value optimization.

    • RVO: (1) the type of the local object is the same as that returned by the function and (2) the local object is what’s being returned.
    • return std::move(w);
      What’s being returned here isn’t the local object w, it’s a reference to w – the result of std::move(w). Returning a reference to a local object doesn’t satisfy the conditions required for the RVOO, so compilers must move w into the function’s return value location.

  • for the situation below, compilers must treat them as rvalues if they’re returned. No need for return std::move(w);

Item 26: Avoid overloading on universal references

  • Overloading on universal references almost always leads to the universal reference overload being called more frequently than expected.
    • Functions taking universal references are the greediest functions in C++. They instantiate to create exact matches for almost any type of argument.
  • Perfect-forwarding constructors are especially problematic, because they’re typically better matches than copy constructors for non-const lvalues, and they can hijack derived class calls to base class copy and move constructors.

Item 27: Familiarize yourself with alternatives to overloading on universal references

  • Alternatives to the combination of universal references and overloading include the use of distinct function names, passing parameters by lvalue-reference-to-const, passing parameters by value, and using tag dispatch.

    • passing parameters by lvalue-reference-to-const, passing parameters by value – Not efficient enough

    • Tag dispatch

  • Constraining templates via std::enable_if permits the use of universal ref‐ erences and overloading together, but it controls the conditions under which compilers may use the universal reference overloads.

  • Universal reference parameters often have efficiency advantages, but they typ‐ ically have usability disadvantages.

    • Some kinds of arguments can’t be perfect-forwarded, even though they can be passed to functions taking specific types.
    • The bad comprehensibility of error messages when clients pass invalid arguments.

Item 28: Understand reference collapsing

  • Reference collapsing rule:

    If either reference is an lvalue reference, the result is an lvalue reference. Otherwise (i.e., if both are rvalue reference) the result is an rvalue reference.

  • Reference collapsing occurs in four contexts:

    • template instantiation(universal reference)

      When an lvalue is passed as an argument, T is deduced to be an lvalue reference. When an rvalue is passed, T is deduced to be a non-reference.

    • auto type generation(universal reference)

    • typedefs and alias declaration(no type deduction)

    • decltype(no type deduction)

  • Universal reference is actually an rvalue reference in a context where 2 conditions are satisfied.

    • Type deduction distinguishes lvalue from rvalues.
    • Reference collapsing occurs.

Item 29: Assume that move operations are not present, not cheap, and not used

  • There are several scenarios in which C++11’s move semantics do you no good:
    • No move operations: The object to be moved from fails to offer move opera‐ tions. The move request therefore becomes a copy request.
    • Move not faster: The object to be moved from has move operations that are no faster than its copy operations.(i.e. std::array, cost as much as copy; std::string small string optimization )
    • Move not usable: The context in which the moving would take place requires a move operation that emits no exceptions, but that operation isn’t declared noex cept.
    • Added: Source object is lvalue: With very few exceptions only rvalues may be used as the source of a move operation.
  • In codwith known types or support move semantics, there is no need for assumptions.
