Public-Key Cryptography Standards, PKCS
Created2020-10-11|Updated2020-10-15
|Post View:

PKCS(Public-Key Cryptography Standards) ，是由 RSA 实验室与其它安全系统开发商为促进公钥密码的发展而制订的一系列标准。

PCKS #1: RSA Cryptography Standard

该部分定义了基于RSA公钥密码系统加密和签名数据的机制。

RSA 密钥生成

密钥生成过程：

  1. 选取$u$个不同的素数 $r_1, r_2, …, r_u$, 各数位数($|r_i|$)相同
  2. $n = \prod_{i=1}^ur_i$，$|n|$的就是密钥长度（modulus位数）。实际应用中，RSA密钥一般是1024位，重要场合则为2048位。
  3. 由CRT(中国剩余定理)可知 Euler函数值$ \phi(n) = \prod_{i=1}^{u}(r_i-1)$
  4. 选择指数$e$, 使得 $gcd(e,\phi(n)) = 1$
  5. 求得指数$d$, 使得 $d = e^{-1}mod\ \phi(n)$
  6. 公钥为$(e,n)$, 私钥为$(d,n)$

$u = 2$时为普通的RSA系统。
$u \ge 3,$ 多素数RSA系统的特点：

  1. 达到相同的modulus位数，所需素数较小，可减少生成密钥的计算量。
  2. 应用CRT可减少解密、签名的计算量。
  3. 素数越多，相同modulus位数所需素因子越小，大数分解越容易，RSA强度越低。

RSA 加解密

加密使用公钥$(e,n)$
$$
c \equiv m^{e} (mod \ n)
$$
解密使用私钥$(d,n)$
$$
c^{d} \equiv m^{ed} \equiv m\ mod \ n
$$

RSA 加密填充

性能原因，RSA 并不用于直接加密长消息，一般用于加密密钥，数据再使用密钥通过如DES和AES的密钥加密技术加密。所以一般来说RSA需要加密的数据比RSA钥模（modulus）要小，需要被加密的数据在加密前先填充到modulus相同的长度。

  1. RSAES-PKCS1-v1_5 padding 简单，安全性较弱
  2. RSAES-OAEP padding（最优非对称填充） 安全性较强

论文中讲述了具体的填充计算方法。

padding的实现与RSA的安全性相关，考虑以下几点：

  1. RSA加密是确定的，即给定一个密钥，特定明文总会映射到特定的密文。攻击者可以根据密文中统计信息获取明文的一些信息。
  2. 填充技术如果比较弱，那么较小的明文和小型公开指数e将易于受到攻击。
  3. RSA有个特性叫做延展性，如果攻击者可以将一种密文转换为另一种密文，儿这种新密文会导致对明文的转换变得可知，这种特性并没有解密明文，而是以一种可预测的方式操纵了明文，比如：银行交易系统中，攻击者根据新密文，直接去修改原密文中金额的数据，可以在用户和接受方无法感知的情况下进行修改。

RSA signature schemes with appendix

RSA 数字签名，最基本的方法是对于整数密文$m\ (m<n)$, 用私钥计算签名$s\equiv m^d\ mod \ n$ , 为验证签名，计算$m’ \equiv s^e\ (mod\ n)$， 若$m' = m$, 则签名有效。

但以上方法只能对长度短于$|n|$的密文生成签名且并不安全。解决方法是，发送方首先用一个哈希函数对报文生成报文摘要（message digest）, 然后用私钥对这个摘要进行加密。

论文中讲述了EMSA-PSS的加密生成报文摘要的方法。


PKCS #3 Diffie-Hellman Key Agreement Standard (Outdated)

Diffie-Hellman密钥协议标准已被其他标准代替。

两种密钥生成方案：

  1. 密钥协商协议：共同产生临时会话密钥，实现保密通信。例如Diffie-Hellman密钥协议。
  2. 密钥传输协议：由一个实体产生密钥.

PKCS #5 Password-Based Cryptography Standard

PKCS#5 provides a general mechanism to achieve an enhanced security for password-based cryptographic primitives, covering derivation functions, encryption schemes, message-authentication schemes and ASN.1 syntax identifying the techniques.

Key-derivation functions

The paper demonstrates a password-based key derivation function PBKDF2.
$$
DK = PBDKF2(P,S,c,dkLen) \
P: password, S: a random salt value, c: iterating count, dkLen: derived Key length
$$

Encryption schemes

The paper demonstrates a encryption scheme PBES2. The underlying encryption scheme $\varepsilon$:
$$
DK = PBDKF2(P,S,c,k)\
C = \varepsilon_{DK}(M)
$$
The decryption can be done similarly.

Message authentication schemes

In a password-based message authentication scheme, the MAC generation operation produces a message authentication code from a message under a password, and the MAC verification verifies the message authentication under the same password.

Example: PBMAC1. The underlying message authentication function $A$:
$$
DK = PBDKF2(P,S,c,k)\
C = A_{DK}(M)
$$
The decryption can be done similarly.

PKCS #7 and RFC3369: Cryptographic Message Syntax(CMS)

PKCS #7，加密消息的语法标准。PKCS#7描述了数字证书、数据加密和数字签名的语法。使用PKCS#7进行数字签名时，结果包含签名证书（一列相关证书撤回列表）和已证明路径上任何其他证书。如果使用PKCS#7加密数据，通常包含发行者的参考消息和证书的序列号，它与用于解密已加密数据的公共密钥相关。

PKCS#7可用于加密两类数据：

（1）基本数据：没有加密的数据，并且包含了的附带物如散列或数字签名。

（2）增强的数据：数据已加密或包含了的附带物，或两者都有。增强的内容以一种形式压缩进另外的内容里面。

PKCS#7标准定义了ContentInfo, 其中有两个成员，ContentType和Content, 其中ContentType包含6种类型

  • data：octet string
  • signed-data
  • enveloped-data
  • digest-data
  • encrypted-data

PKCS #8：Private-Key Information Syntax Standard

PKCS#8 describes a syntax for private-key information, which includes a private key for some public-key algorithm and a set of attributes, and a syntax for encrypted private-key information. A passwrod -based encryption algorithm(e.g., one og those described in PKCS #5) could be used to encrypt the private-key information.


PKCS #9: Selected Object Classes and Attribute Types

PKCS #9 defines two auxiliary object classes, pkcsEntity and naturalPerson. PKCS attributes could be packaged into these two object classes and be exported to other environments. PKCS #9 also defines some new attribute types and matching rules to be used in other PKCS standards.


PKCS #10: Certification Request Syntax Standard

PKCS #10 specifies syntax dor certificate request.

The CertificationRequest should includes $distinguished\ name$,$the \ entity’s\ public\ key $, $a\ set \ of \ attributes$(may be defined in PKCS #9) and be sent to certification authority.


PKCS #11: Cryptographic token Interface Standard

PKCS #11 specifies an API called “Cryptoki” , to devices which hold cryptographic information and perform cryptographic functions.

The goal of Cryptoki is a lower-level programming interface that abstracts the details of the devices, and presents to the application a common model of the cryptographic device, called a “cryptographic token”(“token”).

Crypto defines general data types, objects and functions.

  • Token: a device that stores objects and can perform cryptographic functions.
  • Slot： includes the logic reader of a token.
  • Session: the logic link between token and application. 一个打开的会话能用来执行三大种类的操作：管理操作（如注册），对象管理操作（如在令牌上建立或销毁一个对象）和密码操作（如计算一个消息摘要）
  • Object: an object consists of a set of attributes. Cryptoki recognizes three classes of objects: data, certificates and keys. According to their life time, objects are classified as “token objects” and “session objects”. Cryptoki provides functions for creating, destroying, and copying objects in general, and for obtaining and modifying the value of their attributes.
  • Key: a key object can be a public key, a private key or a secret key. each of these types of keys has subtypes for use in specific mechanisms.
  • Mechanism: a mechanism specifies precisely how a certain cryptographic process is to be performed(encryption algorithm).
  • User type: 2 types: Security Officer(SO) and normal user. The function of SO is to initiate a token and to set the PIN for the normal user. Only normal user has access to private objects in the token.

PKCS #12: Personal Information Exchange Syntax Standard

PKCS #12 describes a transfer syntax for personal identity information, including private keys, certificates, miscellaneous secrets and extensions. Machines support this standard will allow a user to import,export and exercise a single set of personal indentity information.


PKCS 15: Cryptographic Token Information Syntax Standard

PKCS #15 is a standard intended to enable interoperability among components running on various plat- forms (platform neutral), to enable applications to take advantage of products and components from multiple manufacturers (vendor neutral), to enable the use of advances in technology without rewriting application- level software (application neutral), and to maintain consistency with existing, related standards while ex- panding upon them only where necessary and practical.
